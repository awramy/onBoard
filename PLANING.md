## Технологический стек и архитектура MVP сервиса "onBoard"

### 1. Оценка предложенного стека (Node.js + TypeScript + NestJS)

**Вывод: отличный выбор для MVP.**  
Node.js обеспечивает асинхронную неблокирующую модель, идеально подходящую для I/O‑операций (сеть, база данных, AI API). TypeScript добавляет строгую типизацию, что критично для надёжности и поддержки кода. NestJS даёт готовую модульную архитектуру, Dependency Injection, встроенную поддержку различных транспортов (HTTP, WebSocket, микроcервисы) и обширную экосистему.

**Работа с AI моделями:**  
- Через абстрактный `AiProvider` интерфейс можно легко подключать разные SDK (OpenAI, Anthropic, Google Gemini, локальные модели через Ollama и др.).  
- Динамическая смена модели реализуется через конфигурацию (переменные окружения или запись в БД) и фабрику провайдеров.  
- Комьюнити Node.js огромно, есть проверенные библиотеки (например, LangChain.js, Vercel AI SDK), которые унифицируют работу с LLM.  
- Для бесплатных моделей можно использовать Together AI, Groq, или развернуть собственную через Ollama – у всех есть JS SDK.  

**Потенциальные узкие места:**  
- Node.js не предназначен для CPU‑интенсивных вычислений, но в нашем случае вся тяжёлая работа выполняется на стороне AI API.  
- При высоких нагрузках может потребоваться кластеризация, но для MVP достаточно одного инстанса.

| **Очереди**          | Bull (на Redis)            | Для фоновых задач (если понадобятся, например, генерация отчётов) |
| **AI интеграция**    | LangChain.js или прямые SDK (OpenAI, Anthropic) | LangChain упрощает переключение провайдеров и построение промптов; можно начать с OpenAI SDK |

| **Логирование**      | Winston / Pino             | Структурированные логи, интеграция с Nest |
| **Тестирование**     | Jest + Supertest           | Unit и e2e тесты |
| **Документация API** | Swagger (OpenAPI)          | Автоматическая генерация через @nestjs/swagger |

| **Безопасность**     | Helmet, CORS               | Базовая защита |

| **UI компоненты**    | Headless UI / Radix UI     | Доступные примитивы, кастомизация |
| **Чат (реалтайм)**   | (Опционально) Server-Sent Events или WebSocket (Socket.io) | Для будущего стриминга ответов |

**Почему Vite, а не Next.js?**  
MVP не требует SSR и SEO (чат доступен только авторизованным пользователям). Vite проще, быстрее в разработке, легче деплоится как статика. Next.js добавит излишнюю сложность на данном этапе. Однако если в будущем понадобится SSR, миграция возможна.

### 3. Архитектура серверного взаимодействия

#### **Монолитная архитектура (рекомендация для MVP)**

Всё приложение реализуется как единый сервис (монолит) с чётким разделением на модули NestJS. Это ускоряет разработку, упрощает деплой и отладку.

**Структура модулей NestJS:**
```
src/
├── main.ts
├── app.module.ts
├── modules/
│   ├── auth/          # регистрация, логин, JWT
│   ├── users/         # профили, прогресс
│   ├── technologies/  # технологии, уровни, темы (CRUD)
│   ├── questions/     # вопросы, привязка к темам
│   ├── sessions/      # создание сессий, вопросы сессии, ответы
│   ├── progress/      # обновление и чтение прогресса пользователя
│   ├── ai/            # фабрика провайдеров, сервис для общения с моделями
│   └── common/        # guards, interceptors, filters, DTO
```

**Взаимодействие между модулями:** через сервисы и интерфейсы (DI). Все запросы проходят через контроллеры REST API.

**Почему не микросервисы?**  
Микросервисы добавляют сложность: сетевое взаимодействие, согласование контрактов, распределённые транзакции. Для MVP это оверкилл. Монолит легко масштабируется вертикально, а при росте нагрузки можно вынести AI‑сервис в отдельный модуль (с очередями), не переписывая всё.

#### **Схема взаимодействия компонентов**

```
[Frontend (Vite/React)] 
        ⇅ HTTP (REST API)
[Backend (NestJS монолит)]
        ├── Модули: Auth, Sessions, Questions, Progress
        ├── База данных (PostgreSQL) — через Prisma
        ├── Кэш/сессии (Redis) — для rate limiting, кэша ответов AI
        └── AI провайдеры (OpenAI, Anthropic, локальная модель) — через HTTP
```

- **Аутентификация:** JWT токен в заголовке `Authorization`.
- **Запросы к AI:** Backend сам обращается к внешним API (OpenAI и др.), форматирует промпт, получает ответ, парсит оценку и комментарии. Результат сохраняется в БД и возвращается фронту.
- **Динамическая смена модели:** Через конфигурационный файл или запись в БД (например, `ai_config`). При инициализации `AiService` выбирает активного провайдера на основе текущих настроек.

#### **Пример работы AI‑сервиса (фабрика)**

```typescript
// ai/ai-provider.interface.ts
export interface AiProvider {
  generateFeedback(question: string, answer: string): Promise<AiFeedback>;
}

// ai/providers/openai.provider.ts
export class OpenAiProvider implements AiProvider { ... }

// ai/providers/anthropic.provider.ts
export class AnthropicProvider implements AiProvider { ... }

// ai/ai.service.ts
@Injectable()
export class AiService {
  private provider: AiProvider;

  constructor(@Inject(CONFIG_OPTIONS) private config: AiConfig) {
    this.setProvider(config.activeProvider);
  }

  setProvider(providerName: string) {
    switch (providerName) {
      case 'openai':
        this.provider = new OpenAiProvider(this.config.openai);
        break;
      case 'anthropic':
        this.provider = new AnthropicProvider(this.config.anthropic);
        break;
      default:
        throw new Error('Unknown provider');
    }
  }

  async getFeedback(question: string, answer: string) {
    return this.provider.generateFeedback(question, answer);
  }
}
```

#### **Хранение конфигурации провайдера**

Можно хранить в отдельной таблице `ai_config` с поддержкой версионирования, либо в `.env` для простоты (требует перезапуска при смене). Для MVP достаточно `.env` + возможность переключения через переменную окружения.

### 4. Обоснование дополнительных решений

- **Redis**: необходим для эффективного rate limiting (чтобы не перегружать AI API), хранения временных данных (например, состояния сессии при использовании WebSocket) и как брокер очередей (Bull). Даже если очереди не нужны сразу, Redis пригодится.
- **Prisma** vs TypeORM: Prisma даёт строгую типизацию на уровне базы данных и удобный клиент, что снижает количество ошибок. Миграции удобны. Для Nest есть пакет `@prisma/nestjs`.
- **LangChain.js**: может быть полезен для сложных цепочек вызовов, но для MVP часто достаточно прямых вызовов API. Однако если планируется много вариаций промптов и разные модели, стоит рассмотреть LangChain.
- **Аутентификация**: JWT с коротким временем жизни + refresh token (опционально). Пароли хранить с bcrypt.
- **Фронтенд**: выбор Vite + React обусловлен скоростью разработки и богатой экосистемой. Zustand — лёгкий и интуитивный, не требует boilerplate.
- **Деплой**: на старте можно использовать Docker + docker-compose на одной виртуалке. Это позволит поднять всё (Node, Postgres, Redis) в контейнерах.

### Заключение

Предложенный стек (Node.js + NestJS + TypeScript) оптимален для MVP сервиса "onBoard". Он обеспечивает гибкость в работе с AI моделями, быструю разработку и лёгкое масштабирование в будущем. Монолитная архитектура на старте снижает сложность, а продуманная модульность позволит при необходимости выделить части в микросервисы. Дополнительные технологии (PostgreSQL, Redis, Prisma, Vite) дополняют картину и проверены временем.
